# Audio Routing Bugs: Interleaved Buffer Mismatch & RT-Safety Violations

## Summary

Several correctness and real-time-safety bugs were found in the audio processing pipeline
spanning `AudioEngine`, `LiveEffectEngine`, and `LV2Plugin`.

**All 6 bugs have been fixed.** Status noted per entry below.

---

## Bug 1 (Critical) ✓ FIXED: All LV2 audio ports connected to the same interleaved pointer

**File:** `src/LV2Plugin.hpp` — `LV2Plugin::process()`, Step A

LV2 plugins expect separate, non-interleaved `float*` buffers per audio channel.
`AudioEngine::processCb()` correctly builds a single interleaved stereo buffer
(`{L0, R0, L1, R1, …}`), but `LV2Plugin::process()` connects **every** audio port
to that same pointer without de-interleaving:

```cpp
// declared but never used:
uint32_t input_index = 0, output_index = 0;

for (auto& p : ports_) {
    if (!p.is_audio) continue;
    float* target = inputBuffer;   // ← same pointer every time
    if (!p.is_input) target = outputBuffer;
    lilv_instance_connect_port(instance_, p.index, target);
}
```

**Effects:**
- **Mono plugins:** receive a half-speed, L/R-mixed signal covering only `nframes/2` real
  frames; the second half of every output period is silence (unwritten buffer).
- **Stereo plugins:** both input ports share the same interleaved data; both output ports
  immediately overwrite each other — stereo image is completely wrong.

**Fix:** Pre-allocate per-channel scratch buffers (not on the RT thread). De-interleave
`inputBuffer` into them before connecting ports, run the plugin, then re-interleave from
the output scratch buffers back into `outputBuffer`.

**Resolution:** `scratch_in_` / `scratch_out_` per-channel `std::vector<float>` members
added to `LV2Plugin`. Allocated in `initialize()` after port discovery. Step A replaced
with a de-interleave loop + per-port `lilv_instance_connect_port` using the scratch
buffers. A re-interleave step after `lilv_instance_run()` merges the output back into
the interleaved `outputBuffer`. Mono plugins are duplicated to both output channels.

## Bug 2 (Critical) ✓ FIXED: Bypass `memcpy` copies only half the interleaved stereo buffer

**File:** `src/LV2Plugin.hpp` — `LV2Plugin::process()`, bypass path

```cpp
// plugin disabled — bypass:
std::memcpy(outputBuffer, inputBuffer, sizeof(float) * numFrames);
```

`inputBuffer` contains `numFrames × 2` floats (interleaved stereo). Only `numFrames`
floats are copied, leaving the right channel of output stale.

**Fix:** Change to `sizeof(float) * numFrames * 2` (or multiply by the channel count).

**Resolution:** Both the bypass path and the "no audio output ports" passthrough in
`LV2Plugin::process()` now use `sizeof(float) * numFrames * 2`. Fixed as part of the
Bug 1 changes.

## Bug 3 (High) ✓ FIXED: Heap allocation and mutex lock on the RT audio thread

**File:** `src/LV2Plugin.hpp` — `LV2Plugin::process()`, Step B (atom delivery)

```cpp
// Runs on JACK RT thread:
std::vector<uint8_t> payload;              // heap allocation
std::vector<uint8_t> evbuf(event_size);   // heap allocation
...
std::lock_guard<std::mutex> lock(p.atom_state->ui_to_dsp_mutex); // can sleep
payload = p.atom_state->ui_to_dsp;        // vector copy → heap allocation
```

Both `std::vector` construction/copy and `std::mutex::lock()` can block the RT thread
(memory allocator contention, priority inversion). On a loaded system this produces xruns.

**Fix:**
- Replace the mutex with a lock-free ringbuffer (an `lv2_ringbuffer_t` is already present
  in the codebase) for UI→DSP atom messages.
- Use a pre-allocated, fixed-size staging buffer; write into it without dynamic allocation.

**Resolution:** `AtomState` now uses a second `lv2_ringbuffer_t* ui_to_dsp` (64 KB) in
place of the mutex + vector. `send_path_parameter` writes the full `LV2_Atom` (header +
body) to this ringbuffer in one call. `process()` Step B drains the ringbuffer in a
`while` loop using the pre-allocated `atom_staging_buf_` member — zero heap allocations
and zero mutex locks on the RT thread. Multiple queued messages per cycle are now handled
correctly.

## Bug 4 (Medium) ✓ FIXED: `blockSize` default mismatch between `LiveEffectEngine` and `AudioEngine`

**Files:** `src/LiveEffectEngine.h`, `src/linux/AudioEngine.h`

```cpp
// LiveEffectEngine.h:
int blockSize = 4096;

// AudioEngine.h default:
std::atomic<int32_t> blockSize_{ 1024 };
```

A plugin loaded before `AudioEngine::start()` is called (e.g. from a saved preset) will
be instantiated with `max_block_length = 4096`. If the system JACK block size is 512 or
1024, the options negotiation (`LV2_BUF_SIZE__maxBlockLength`) advertises a value 4× too
large. This wastes plugin-internal buffer memory and may mislead plugins that size their
internal delay lines from that option.

**Fix:** Initialise `LiveEffectEngine::blockSize` to `0` or a sentinel, and always call
`AudioEngine::start()` before loading plugins, or re-instantiate plugins after the actual
block size is known.

**Resolution:** `LiveEffectEngine::blockSize` initialised to `0`. `addPlugin()` now
rejects calls with `blockSize <= 0` and logs an error. `MainWindow::onAddPlugin()` guards
against calling `addPlugin` before `AudioEngine::State::Running`, surfacing
`"Start the engine before loading plugins"` in the status bar.

## Bug 5 (Low) ✓ FIXED: `buf_minBlockLength` always set equal to `buf_maxBlockLength`

**File:** `src/LV2Plugin.hpp` — `init_instance()`

```cpp
{ LV2_OPTIONS_INSTANCE, 0, urids_.buf_minBlock, sizeof(uint32_t), urids_.atom_Int, &max_block_length_ },
{ LV2_OPTIONS_INSTANCE, 0, urids_.buf_maxBlock, sizeof(uint32_t), urids_.atom_Int, &max_block_length_ },
```

JACK can deliver blocks smaller than the declared max (e.g. at transport start/stop).
Plugins that check `buf_minBlock` will assume a fixed block size and may mishandle
variable-length callbacks.

**Fix:** Either advertise `buf_minBlock = 1` (safest) or the actual minimum JACK block
size if known.

**Resolution:** `init_instance()` now declares `static const uint32_t kMinBlockLength = 1`
and passes `&kMinBlockLength` for the `buf_minBlock` option. `buf_maxBlock` and
`buf_nominalBlock` still use `max_block_length_` (the real JACK buffer size).

## Bug 6 (Low) ✓ FIXED: Interleaved work buffers not reallocated on JACK buffer-size change

**File:** `src/linux/AudioEngine.cpp` — `Impl::bufferSizeCb` / `start()`

```cpp
// start(): pre-allocate once:
const size_t bufSamples = static_cast<size_t>(bs) * 2 * 4; // ×4 "guard"

// bufferSizeCb: only updates the atomic, no realloc:
owner_ptr->blockSize_.store(static_cast<int32_t>(nframes), std::memory_order_relaxed);
```

If JACK raises its block size above `bs × 2` frames at runtime, `processCb` will write
beyond the pre-allocated buffer (buffer overflow). The `×4` multiplier is a fragile
guard, not a proper fix.

**Fix:** In `bufferSizeCb`, signal the main thread (via `g_idle_add` or an atomic flag)
to reallocate the interleaved buffers to `nframes * 2`, then let `processCb` wait until
reallocation is complete before processing.

**Resolution:** Added `std::atomic<uint32_t> pendingBufSize_` to `Impl`. `bufferSizeCb`
(now receiving `Impl*` instead of `AudioEngine*`) stores the new size and schedules a
`g_idle_add` callback that reallocates `interleavedIn`/`interleavedOut` to `nframes * 2`
on the GTK main thread and clears `pendingBufSize_` with a release store. `processCb`
checks `pendingBufSize_` with an acquire load at entry and falls back to a direct
`memcpy` from JACK port buffers while reallocation is in flight. The fragile `×4`
over-allocation guard has been removed.
